**11\_Evidence·Reviewer·Correction 정책 문서 v1.0**

**1\. 문서 목적**

이 문서의 목적은 문화강국네트워크 AI홈페이지에서 **무엇을 근거로 삼고, 누가 검토하며, 어떤 수정·정정 절차로 신뢰를 유지할 것인지**를 정책 수준으로 고정하는 데 있다.  
이 문서는 단순한 “출처 표기 가이드”가 아니다. 이 문서는 아래를 함께 규정한다.

*   Evidence의 정의와 유형
*   Reviewer의 유형과 책임
*   Boundary / Disclaimer 기준
*   Update와 ChangeLog 기준
*   Correction 요청과 반영 절차
*   공개 승인 조건
*   SLA와 운영 책임

즉, 이 문서는 문화강국네트워크 AI홈페이지의 **Trust Layer 운영 헌장**이다. B-SSoT 공통 구조가 Trust Layer를 별도 층으로 두고, 운영형 정답 인프라의 핵심으로 규정하기 때문에 이 문서는 선택이 아니다.

**2\. 최상위 원칙**

**2-1. 신뢰는 보조 요소가 아니라 구조다**

문화강국네트워크 AI홈페이지에서 신뢰는 “좋은 느낌”이 아니라 구조적으로 보여야 한다.  
즉, 사용자는 각 AnswerCard, Story, Event, Video, ArchiveDocument에서 최소한 다음을 볼 수 있어야 한다.

*   누가 썼는가
*   누가 검토했는가
*   어떤 자료를 근거로 하는가
*   마지막 업데이트는 언제인가
*   필요한 경우 어떤 한계와 경계가 있는가
*   수정되었다면 어디가 수정되었는가

이 원칙은 AI홈페이지 공통 원칙인 **Trust always visible**과 정확히 일치한다.

**2-2. Media 패밀리의 신뢰는 편집권·취재·전문성·출처 체인에서 나온다**

문화강국네트워크는 Media 패밀리다. Media 패밀리의 신뢰 방식은 **편집권, 취재, 전문성, 출처 체인**이며, reviewer 축은 **editor / reporter / expert**로 본다. 따라서 병원이나 금융처럼 안전 고지 중심이 아니라, **편집 책임과 출처 연쇄가 드러나는 구조**가 핵심이다.

**2-3. 공개는 Review 이후에만 가능하다**

Draft에서 바로 Public으로 가지 않는다.  
hanjunho.ai 운영 플레이북도 **Draft → Review → Public**을 기본 흐름으로 두고, 공개 상태는 엄격하게 관리해야 한다고 규정합니다. 문화강국네트워크도 동일하게 적용한다.

**2-4. 기록은 남기고, 정정은 숨기지 않는다**

수정과 정정은 실패의 흔적이 아니라 운영형 정답 인프라의 정상 동작이다.  
B-SSoT는 ChangeLog/TrustLog 기반 운영형 구조이며, Trust Layer에 Correction Path가 포함된다. 따라서 중요한 수정은 changelog로 남겨야 한다.

**3\. Evidence 정책**

**3-1. Evidence의 정의**

Evidence는 콘텐츠의 핵심 주장, 정의, 일정, 사실관계, 해설을 **검증 가능하게 만드는 자료**다.  
문화강국네트워크에서 Evidence는 단순 링크가 아니라, 정답·해설·행사·자료를 연결하는 **근거 레이어**로 작동해야 한다. B-SSoT 공통 구조가 최소 정답 단위에 Evidence를 필수로 포함하는 이유도 여기에 있다.

**3-2. Evidence 유형**

문화강국네트워크에서는 Evidence를 아래 5종으로 분류한다.

**A. 공식 내부 자료**

*   이사장 발간사
*   편집장 노트
*   공식 소개문
*   내부 기준 문서
*   공식 행사 기획 개요

**B. 행사 기록 자료**

*   발제문
*   자료집
*   행사 프로그램
*   사진 기록
*   행사 영상
*   녹취 또는 요약 기록

구성안이 “토론회 개요 및 발제문 아카이빙”과 “문화행사 사진 위주의 개요 안내”를 명시하므로, 이 유형은 핵심 evidence 축이다.

**C. 외부 공적 자료**

*   정부/기관 발표 자료
*   법안/정책 문서
*   공공 통계
*   공식 기관 보고서

**D. 언론·전문가 자료**

*   주요 기사
*   인터뷰 원문
*   전문 기고문
*   학술/연구 자료

**E. 내부 정본 객체**

*   AnswerCard
*   TopicSummary
*   Story
*   Event
*   ArchiveDocument

이 사이트 안에서 이미 검수와 근거를 거친 정본 객체도 다른 객체의 evidence layer 일부가 될 수 있다.

**3-3. Evidence 최소 기준**

**AnswerCard**

*   최소 1개 이상의 명시적 evidence 필수
*   가능하면 2개 이상 권장
*   외부 또는 행사/자료형 evidence 최소 1개 포함 권장

**Story**

*   Based On Answers 최소 1개 필수
*   직접 evidence 또는 관련 자료 연결 최소 1개 필수
*   Agenda Essay처럼 해설형 글은 외부 공적 자료 또는 행사 기록 연결 권장

**Event**

*   Event Summary 외에 행사 개요/프로그램/발제문/사진/영상 중 최소 1개 이상 연결 권장
*   종료 후 아카이브화 단계에서는 자료 또는 영상 중 최소 1개 확보 권장

**Video**

*   원출처, 연관 행사 또는 연관 Story 중 최소 1개 필수

**ArchiveDocument**

*   문서 자체가 evidence이지만, 무엇의 evidence인지 연결 객체가 필수

hanjunho.ai도 Answer Card는 근거 링크가 전혀 없는 상태로 공개하지 않는 것을 원칙으로 하고, 기사형 해설도 기사 출처와 관련 카드 연결을 상단에 두도록 규정합니다.

**3-4. Evidence 표기 방식**

Evidence는 “자료 있음”처럼 추상적으로 적지 않는다. 아래 형식을 기본으로 한다.

*   자료 제목
*   자료 유형
*   발행 주체 또는 출처
*   날짜(가능 시)
*   링크 또는 파일
*   연결 이유(선택)

예:

*   토론회 발제문 – K-문명의 가능성 | 발제문 | 문화강국네트워크 | 2026-03-10
*   문화정책 브리핑 자료집 | 자료집 | 문화강국네트워크 | 2026-03-12

**4\. Reviewer 정책**

**4-1. Reviewer의 정의**

Reviewer는 해당 객체가 **표현상·구조상·근거상 공개 가능한 수준인지 검토하는 책임 주체**다.  
Reviewer는 작성자를 대체하지 않는다. 작성은 생산, review는 검증이다.

**4-2. Reviewer 유형**

문화강국네트워크는 Media 패밀리이므로 reviewer 축을 아래 3종으로 둔다.

**A. Editor**

*   편집 구조, 제목, 문장, 맥락, 연결 구조를 검토
*   문강 RIO 글, TopicSummary, 홈 큐레이션에 중요

**B. Reporter / Curator**

*   기록 정확성, 행사 정리, 자료 맥락화를 검토
*   Event, ArchiveDocument, Event Review에 중요

**C. Expert**

*   개념 정의, 정책 해석, 전문성 관련 문장을 검토
*   Agenda Essay, Policy Brief, K-문명/문화정책/지역문화 관련 정답카드에 중요

이 축은 Media 패밀리의 reviewer 축을 그대로 적용한 것이다.

**4-3. 역할 분리 원칙**

하나의 객체에는 최소 아래 역할을 구분한다.

*   Author: 초안 작성
*   Editor: 문장/구조/큐레이션 검토
*   Reviewer: 공개 승인 관점의 최종 검토

소규모 운영에서는 같은 사람이 두 역할을 겸할 수 있지만, 공개 화면에는 가능한 한 작성과 검수를 분리 표기하는 편이 좋다.

**4-4. Reviewer 필수 규칙**

**필수 공개 대상**

*   AnswerCard
*   Story
*   TopicSummary
*   Event
*   Video
*   ArchiveDocument(권장)

**공개 금지 조건**

*   Reviewer 없음
*   Reviewer는 있으나 역할이 불명확함
*   Reviewer가 있는데 Last Updated가 없음

hanjunho.ai 운영 플레이북은 Reviewer를 카드와 브리핑 공개의 기본 기준으로 두고, Reviewer 누락 금지를 긴급 대응 원칙에서도 강조합니다.

**5\. Boundary / Disclaimer 정책**

**5-1. Boundary의 정의**

Boundary는 해당 콘텐츠가 **무엇을 말하고 있고 무엇을 아직 말하지 않는지**, 그리고 **어디까지가 공식 입장이고 어디부터가 해설/해석인지**를 설명하는 문장이다.

**5-2. Boundary가 특히 필요한 경우**

*   K-문명, 문화강국처럼 선언으로 오해될 수 있는 개념형 글
*   인터뷰이 발언을 담은 글
*   진행 중인 행사나 아직 정리 중인 이슈
*   외부 자료를 요약·해석한 글
*   지역 사례를 일반화하는 글

실제 K-문명 글도 K-문명을 우월성 선언이 아니라 분석적 이름으로 한정하고, 자부심이 배타적 우월성으로 흐르면 안 된다고 경계합니다. 이처럼 Boundary는 문강 RIO에서 특히 중요합니다.

**5-3. Boundary 문구 예시**

*   **이 글은 문화강국네트워크의 공식 입장을 바탕으로 작성된 해설입니다.**
*   **이 인터뷰의 견해는 인터뷰이의 발언을 바탕으로 정리되었으며, 단체의 전체 공식 입장과는 구분될 수 있습니다.**
*   **이 글은 개념을 설명하기 위한 분석적 정리이며, 선언적 구호로 읽혀서는 안 됩니다.**
*   **이 자료는 공개 가능한 범위에서 정리된 기록입니다. 이후 자료 보강에 따라 수정될 수 있습니다.**

**6\. Update / ChangeLog 정책**

**6-1. Update의 정의**

Update는 내용의 핵심이 바뀌지 않더라도 **정보 보강, 자료 추가, 링크 보완, 최신성 반영**이 이루어진 경우를 뜻한다.

**6-2. ChangeLog의 정의**

ChangeLog는 사용자가 알아야 할 수준의 중요한 수정 사항을 기록하는 로그다.

**6-3. Update와 ChangeLog의 구분**

**Update만 필요한 경우**

*   링크 수정
*   오탈자 정정
*   이미지 교체
*   참고 자료 1개 추가

**ChangeLog가 필요한 경우**

*   핵심 정의가 바뀐 경우
*   날짜/이름/장소/행사 내용 오류 수정
*   발제문/영상/자료가 추가되어 페이지의 정보 수준이 크게 달라진 경우
*   공식 입장, 해설 방향, boundary 문구가 바뀐 경우

B-SSoT는 ChangeLog/TrustLog 기반 운영형 구조라고 명시되어 있으므로, 문화강국네트워크도 중요한 수정은 구조적으로 남겨야 한다.

**6-4. 표기 규칙**

*   화면에는 최종 업데이트를 항상 노출
*   중요한 수정이 있으면 변경 이력 링크 노출
*   Event 종료 후 자료/영상이 추가되면 업데이트일을 반드시 갱신

**7\. Correction 정책**

**7-1. Correction의 정의**

Correction은 사실관계, 출처, 시점, 역할, 행사 기록, 인물 정보 등에서 오류가 발견되어 수정하는 절차다.

**7-2. Correction 발생 유형**

*   인물명·직함·기관명 오류
*   행사 일정/장소/프로그램 오류
*   발제문/자료 링크 오류
*   주장과 근거의 불일치
*   인터뷰 인용 오류
*   단체 공식 입장과 해설문 표현의 혼선

**7-3. Correction 처리 단계**

1.  **접수**
    *   문의 폼
    *   내부 운영자 발견
    *   작성자/검수자 발견
2.  **분류**
    *   단순 표기 오류
    *   핵심 사실 오류
    *   해석/경계 문제
    *   링크/자료 문제
3.  **검토**
    *   Editor 또는 담당 운영자 1차 검토
    *   필요 시 Expert reviewer 확인
4.  **조치**
    *   수정
    *   temporary hidden
    *   change log 추가
5.  **재공개**
6.  **기록 보존**

이 단계는 B-SSoT의 운영 루프인 Validation → Publish → Measure → Root Cause → Fix-It → Re-Publish와 같은 철학을 콘텐츠 수정에 적용한 것이다.

**7-4. Correction 요청 페이지/채널**

문화강국네트워크는 최소 아래 채널을 가진다.

*   정정·문의 이메일 또는 폼
*   협력·문의 채널 내 정정 선택지
*   운영자 내부 수정 요청 루트

**7-5. 화면 문구 예시**

*   **정정 요청이 필요하면 문의 채널을 통해 알려주세요. 확인 후 반영하겠습니다.**
*   **중요한 수정은 변경 이력에 반영됩니다.**

**8\. 공개 승인 정책**

**8-1. 공통 공개 승인 기준**

모든 핵심 공개 객체는 최소 아래를 만족해야 한다.

*   역할이 분명하다
*   Author 또는 Owner가 있다
*   Reviewer가 있다
*   Last Updated가 있다
*   최소한의 Evidence 또는 관련 자료가 있다
*   Related Object가 있다
*   CTA가 있다
*   Public Status가 적절하다

hanjunho.ai 운영 플레이북도 공개 직전 핵심 기준으로 구조, 출처, Reviewer, 메타, 상태값, 샘플 잔존 여부를 점검해야 한다고 정리합니다.

**8-2. 객체별 공개 승인 기준**

**AnswerCard**

*   질문 1개
*   Direct Answer
*   Evidence 1개 이상
*   Reviewer
*   Related Story 1개 이상 권장
*   Public Status = Review 통과 후만 Public

**Story**

*   Based On AnswerCard 1개 이상
*   Why This Matters
*   Reviewer
*   관련 자료 또는 Event/Video/Archive 최소 1개 이상
*   범용 칼럼형 글 금지

hanjunho.ai도 “Related Cards가 전혀 없는 글”, “기사 출처가 불명확한 글”, “범용 칼럼형 글”을 공개 금지 사례로 들고 있습니다. 문화강국네트워크도 동일한 원칙을 적용하는 것이 맞습니다.

**Event**

*   Event Summary
*   일정/장소
*   상태값
*   관련 Topic/Answer 연결
*   자료 또는 후속 Story 중 최소 1개 이상 권장

**Video**

*   Summary
*   Embed URL
*   Event/Story 중 최소 1개 연결

**ArchiveDocument**

*   파일 또는 URL
*   연결 객체 1개 이상
*   문서 유형 명시

**9\. 운영 역할 분담**

hanjunho.ai 운영 플레이북은 운영 총괄, 콘텐츠 기획·편집, 근거·리서치, 검수, CMS 운영, 기술 지원 역할을 구분합니다. 문화강국네트워크도 같은 방식이 적합합니다.

**9-1. 운영 총괄**

*   공개 승인
*   민감 이슈 최종 판단
*   정정 우선순위 판단

**9-2. 콘텐츠 기획·편집**

*   AnswerCard / Story 초안
*   연결 구조 설계
*   질문 구조화

**9-3. 근거·리서치 담당**

*   출처 수집
*   발제문/자료/사진/링크 정리
*   행사 기록 보강

**9-4. 검수 담당**

*   표현의 정확성
*   구조의 명료성
*   개념의 적절성
*   공개 승인 전 최종 검토

**9-5. CMS 운영 담당**

*   상태값 변경
*   Featured 지정
*   메타 보완
*   발행 후 수정 반영

**10\. SLA 권장안**

**10-1. 일반 오탈자/링크 수정**

*   접수 후 3영업일 이내 확인
*   5영업일 이내 수정 권장

**10-2. 사실관계 오류**

*   접수 후 1영업일 이내 내부 확인 시작
*   3영업일 이내 수정 또는 임시 hidden 여부 결정

**10-3. 행사 종료 후 기록 반영**

*   행사 종료 후 7일 이내: Event Summary / 사진 1차 반영 권장
*   행사 종료 후 14일 이내: 자료집/발제문/영상 반영 권장

**10-4. 정답카드/스토리 최신성 점검**

*   핵심 아젠다 카드: 분기 1회 검토 권장
*   행사 연동 카드/스토리: 이벤트 직후 검토 권장

**11\. QA / lint 규칙**

**11-1. Evidence lint**

*   Evidence 0개
*   출처 제목 없음
*   링크 깨짐
*   무엇의 근거인지 불명확

**11-2. Reviewer lint**

*   Reviewer 없음
*   Reviewer 역할 불명확
*   화면 표기와 schema/cms 불일치

**11-3. Update lint**

*   Last Updated 없음
*   오래된 날짜 그대로 방치
*   ChangeLog가 필요한 수정인데 기록이 없음

**11-4. Correction lint**

*   정정 문의 채널 없음
*   수정했지만 변경 이력 없음
*   오류가 반복되는데 root cause 정리가 없음

hanjunho.ai QA-UAT 문서도 title/meta/schema 못지않게 **상태값, 용어 일관성, 업데이트일, 샘플/테스트 콘텐츠 노출**을 중요한 품질 기준으로 둡니다. 문화강국네트워크도 동일한 방식으로 QA해야 합니다.

**12\. 문화강국네트워크 적용 예시**

**12-1. 토론회 Event 페이지**

*   Evidence: 행사 개요, 프로그램, 발제문 PDF, 사진
*   Reviewer: 편집 책임자 또는 행사 기록 담당자
*   Update: 발제문 추가 시 날짜 갱신
*   Correction: 연사명/직함/일정 오류 수정 시 changelog 반영

**12-2. 문강 RIO Story – K-문명의 가능성**

*   Evidence: 관련 발제문, 관련 토론회, 내부 정답카드, 외부 기사/공적 자료
*   Reviewer: 편집장 + 주제 검토자
*   Boundary: “선언”이 아니라 “분석적 이름”이라는 경계 명시
*   Correction: 개념 오해를 불러올 표현이 있으면 수정 이력 남김

실제 원고가 K-문명을 선언이 아닌 분석 개념으로 한정하므로, reviewer와 boundary가 특히 중요합니다.

**12-3. 회원가입 안내 AnswerCard**

*   Evidence: 회원 혜택/절차 공식 안내
*   Reviewer: 운영 책임자
*   Update: 혜택 변경 시 날짜 갱신
*   Correction: 잘못된 안내 문구는 즉시 수정

**13\. 최종 결론**

이 정책 문서의 핵심은 단순합니다.  
문화강국네트워크 AI홈페이지에서 Evidence, Reviewer, Update, ChangeLog, Correction은 선택 장식이 아니라 **정답 인프라의 기본 계약**입니다. 홈페이지 구성안이 토론회 개요, 발제문 아카이빙, 문화행사 기록, 문화예술정보, 회원가입 안내를 함께 요구하고, B-SSoT가 최소 정답 단위를 질문+정답+근거+검수+업데이트+행동으로 정의하며, hanjunho.ai 운영 체계가 Reviewer, 출처, 상태값, Review 단계, 공개 후 수정 반영을 핵심 운영 규칙으로 고정한 점을 종합하면, 문화강국네트워크는 \*\*“무엇을 말하느냐” 못지않게 “무엇을 근거로, 누가 검토해, 어떻게 수정하며 말하느냐”\*\*를 시스템으로 보여줘야 합니다. 그게 이 AI홈페이지의 신뢰 구조입니다.
