**33\_프론트엔드 컴포넌트-페이지 매핑표 v1.0**

**1\. 문서 목적**

이 문서는 퍼블릭 사이트와 Admin Console의 각 페이지가 어떤 **컴포넌트 조합**으로 구성되고, 어떤 데이터 소스/API에 바인딩되며, 어떤 상태(empty/loading/error)를 가져야 하는지 매핑하는 문서다.

**2\. 설계 원칙**

*   page template 중심
*   component 재사용 우선
*   퍼블릭/어드민 shell 분리
*   모든 상세 페이지는 trust block 포함
*   모든 리스트 페이지는 filter/search/sort 표준 패턴 사용
*   loading / empty / error를 별도 정의
*   Admin 페이지는 list/detail/form/board/queue 유형으로 분류

B-SSoT는 Visual Canonical Objects와 Trust Layer를 3층 구조의 핵심으로 보고, Answer-first와 Graph linking을 UI 원칙으로 둔다. 프론트 컴포넌트도 이 구조를 따라야 한다.

**6\. 데이터 바인딩 규칙**

*   퍼블릭 상세는 public read API만 사용
*   Admin 상세는 admin object API + related ops API 결합
*   trust/meta/feed/sitemap snapshot은 sidebar or lower panel로 분리
*   상태 전이 액션은 optimistic UI 금지 권장
*   destructive action은 confirm modal 필수

**7\. 최종 결론**

33번 문서는 31번 API 계약과 29번 페이지 구현 명세를 실제 UI 조립 단위로 연결한다. 핵심은 문화강국네트워크 퍼블릭 사이트를 허브/상세 템플릿으로, Admin Console을 리스트/상세/폼/보드/큐 패턴으로 표준화하는 것이다. 이를 통해 문강 RIO, 정답카드, 행사·자료, 회원·참여, Feed·Sitemap·Observatory가 재사용 가능한 컴포넌트 체계 위에서 구현된다.
